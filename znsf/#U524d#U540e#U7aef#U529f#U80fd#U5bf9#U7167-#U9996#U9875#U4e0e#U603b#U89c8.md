# 前后端功能对照（概要版）

> 说明：本文件按“页面维度”梳理。每个页面只列出关键前端功能点，以及后端需要提供的接口和数据职责，力求简洁明了。  
> 先从首页开始，后续再逐页补充 AI 看板、报表中心、患者管理、随访管理等。

---

## 一、首页 / 数据总览（路由：`/home`，组件：`views/home/Index.vue`）

### 1. 顶部统计卡片区

**前端功能点**
- 展示当前：
  - 在管患者总数及环比变化
  - 本周随访完成率、本周完成随访人数
  - 异常告警处理中数量、其中红色告警数量

**后端需要做的事**
- 提供首页汇总统计接口（示例）：
  - `GET /api/home/stats`
- 返回的核心字段建议：
  - `totalPatients`：在管患者总数
  - `totalPatientsChange`：较上期变化描述（字符串即可，如 `+12人` / `-5人`）
  - `weekFollowRate`：本周随访完成率（0–1 小数）
  - `weekFollowDone`：本周完成随访人数
  - `processingAlerts`：当前处理中告警总数
  - `redAlerts`：其中红色/高危告警数量

---

### 2. 中部图表区：风险分层趋势 / 告警与随访趋势 / 风险占比饼图

**前端功能点**
- 折线图 1：按时间（通常按月）展示：
  - 高危 / 中危 / 低危患者数量变化趋势
- 折线图 2：按时间展示：
  - 告警次数、随访次数变化趋势
- 饼图：当前高危 / 中危 / 低危患者占比

**后端需要做的事**
- 提供月度（或按时间）汇总数据接口（示例）：
  - `GET /api/report/month-summary`
- 建议返回为一个数组，每条代表一个时间点，核心字段：
  - `month`：时间点（如 `2025-06`）
  - `highRisk` / `midRisk` / `lowRisk`：对应时间点高/中/低危人数
  - `alerts`：告警次数
  - `followups`：随访次数
- 要求：
  - 时间顺序最好按从旧到新排列，方便前端直接画图
  - 若暂时数据不足，也可只返回最近几个月的数据

---

### 3. 底部：待处理高危告警清单 & 近期证候加重患者

**前端功能点**
- 卡片 1：“待处理高危告警清单”：
  - 展示少量高危告警摘要（患者姓名、主要风险因素、告警时长、责任医生）
- 卡片 2：“近期证候加重患者”：
  - 展示近期证候加重的患者名单（姓名、变化描述、时间）
- 点击卡片可跳转到对应模块（告警中心 / 患者档案等）

**后端需要做的事（推荐）**
- 高危告警清单接口（示例）：
  - `GET /api/alert/home-high-risk-list`
  - 字段建议：
    - `patientName`：患者姓名
    - `riskFactors`：主要风险因素描述
    - `durationText`：告警持续时长（前端可直接展示文本）
    - `doctorName`：责任医生
- 证候加重患者接口（示例）：
  - `GET /api/patient/recent-syndrome-worsen`
  - 字段建议：
    - `patientName`：患者姓名
    - `changeDesc`：证候变化描述
    - `timeText`：时间描述（如 `2小时前`、`昨天`）
- 跳转只依赖路由，不额外需要后端支持；若希望精确联动，可以在以上接口中增加：
  - `patientId` / `riskId` / `alertId` 等，用于后续页面精准定位。

---

### 4. 顶部统计卡片的跳转联动

**前端功能点**
- 点击不同统计卡片，跳转到对应的业务页面，例如：
  - 在管患者总数卡片 → 患者列表页 `/patient`
  - 本周随访完成率卡片 → 主动随访工作台 `/followup/workbench`
  - 异常告警处理中卡片 → 告警中心 `/alert`

**后端需要做的事**
- 这些跳转本身不需要额外接口，只依赖上面已经提到的各业务模块接口：
  - 患者列表：患者管理模块的 `GET /api/patient/summary`
  - 随访工作台：随访模块的随访任务列表接口
  - 告警中心：告警模块的告警列表接口
- 只要各模块的列表接口支持基本筛选参数（风险等级、时间范围、状态等），即可满足首页跳转后的使用场景。

## 二、AI 分析看板（路由：`/home/board`，组件：`views/report/Board.vue`）

> 说明：本页核心是“本月概览 + 趋势图 + 疾病统计报表 + 季节因素分析”。所有数据建议统一由报表接口 `/api/report/board` 提供，规则类信息由 `/api/system/rules` 提供。

### 1. 顶部三张卡片：本月数据统计 / 患者风险分层 / AI 模型性能

**前端功能点**
- 卡片 1：“本月数据统计”
  - 显示最新统计月份、本月新增患者、本月异常告警、本月随访任务数。
- 卡片 2：“患者风险分层”
  - 显示当前高危 / 中危 / 低危患者数量及合计。
- 卡片 3：“AI 模型性能指标”
  - 显示风险预测准确率、告警识别率、告警趋势变化（上升/下降）。

**后端需要做的事**
- 提供报表总览接口（示例）：
  - `GET /api/report/board`
- 建议字段：
  - `latestMonth`：最新统计月份（如 `2025-06`）
  - `latestNewPatients`：本月新增患者数
  - `latestAlerts`：本月异常告警数
  - `latestFollowups`：本月随访任务数
  - `latestHighRisk` / `latestMidRisk` / `latestLowRisk`：当前高/中/低危患者数
  - `latestAuc` / `latestF1`：模型 AUC、F1（0–1 小数，前端转为百分比）
  - `alertChangeRate`：告警数量较上月的变化比例（小数，正负皆可）

---

### 2. 中部趋势图：告警 & 随访趋势 / 高危 & 其他人群趋势

**前端功能点**
- 折线图 1：时间维度上展示告警次数、随访次数趋势。
- 折线图 2：时间维度上展示高危人数 vs 其他风险层级人数趋势。

**后端需要做的事**
- 继续复用 `GET /api/report/board`，在同一返回体中增加数组字段：
  - `months: string[]`：时间点列表（如月份数组）
  - `alertsArr: number[]`：各时间点告警次数
  - `followArr: number[]`：各时间点随访次数
  - `highArr: number[]`：各时间点高危人数
  - `otherArr: number[]`：各时间点其他（中危+低危）人数
- 要求：
  - 数组长度一致，一一对应同一批 `months`
  - 至少提供最近若干个时间点（如 6~12 个），以便画出趋势。

---

### 3. 病种统计分析报表（病种与康养方案效果概览）

**前端功能点**
- 按“病种”维度汇总展示：
  - 患者数、稳定率 / 改善率 / 恶化率
  - 典型患者画像（年龄段、阶段、体质、合并症、生活方式）
  - 不同康养方案（饮食、运动、用药、中医调理等）的效果对比
  - 核心加重因素（如熬夜、情绪、季节、依从性差等）
  - 面向医生的简要建议要点。
- 支持前端本地筛选（按病种/患者数/稳定率区间）和分页（前端分页）。

**后端需要做的事**
- 仍由 `GET /api/report/board` 返回一个病种数组字段，例如：
  - `diseaseAnalysis: DiseaseAnalysisData[]`
- 单条建议字段：
  - `disease`：病种名称
  - `patientCount`：在管患者数
  - `stableRate` / `improvementRate` / `deteriorationRate`：稳定/改善/恶化比例（0–1）
  - `carePlanEffectiveness`：各康养方案效果列表，包含：
    - `planName`、`planType`、`patientCount`、`effectiveness`（0–100 分）等
  - `deteriorationFactors`：加重因素列表，包含：
    - `factorName`、`impactLevel`（高/中/低）、`patientCount`、`description`
  - `typicalProfiles`：典型患者画像列表，包含：
    - `ageRange`、`diseaseStage`、`constitution`、`comorbidities[]`、`lifestyle[]`、`typicalCount`
  - `recommendations`：医生建议要点字符串数组。
- 要求：
  - 不需要一次性非常精细，哪怕先只提供病种名称、患者数和几项简单比例，前端也能先画基础报表；后续逐步丰富字段即可。

---

### 4. 季节因素影响分析

**前端功能点**
- 展示若干“季节 / 天气因素”卡片，每个卡片包括：
  - 因素名称（寒冷季节、梅雨、高温、雾霾等）
  - 权重或影响描述
  - 主要受影响的病种说明（例如：寒冷 → 心脑血管疾病风险上升）。

**后端需要做的事**
- 由系统规则接口提供季节因素配置（示例）：
  - `GET /api/system/rules`
  - 字段：
    - `seasonRules: { factorName: string; weightDesc: string }[]`
- 字段含义：
  - `factorName`：因素名称（前端用来决定展示文案、并根据关键字推断主要影响病种）
  - `weightDesc`：影响描述（可以是自然语言，如“显著增加心血管事件风险”）。
- 可选增强：
  - 若后端愿意直接给出影响病种，也可以再加字段：
    - `affectedDiseases: string[]`，前端直接展示，不再依赖本地推断。

## 三、报表中心（路由：`/home/export`，组件：`views/report/Export.vue` 或类似）

> 说明：本页核心是“选择时间 / 条件 → 导出不同主题报表（患者、随访、告警、病种分析等）”。前端只负责收集筛选条件、触发导出；后端负责生成文件并返回下载链接或文件流。

### 1. 报表类型选择与筛选条件

**前端功能点**
- 提供报表类型下拉或单选项，例如：患者清单、随访记录、告警记录、病种分析汇总等。
- 提供通用筛选条件：
  - 时间范围（开始日期、结束日期）
  - 机构 / 科室（若有多机构、多科室）
  - 医生 / 随访人员（可选）
  - 风险等级（高/中/低）等。

**后端需要做的事**
- 提供统一导出接口（示例）：
  - `POST /api/report/export`
- 请求体建议字段：
  - `reportType`：报表类型（如 `patientList` / `followup` / `alert` / `diseaseSummary`）
  - `startDate` / `endDate`：时间范围
  - `orgId` / `deptId` / `doctorId` / `staffId`：机构、科室、医生、随访人员（可选）
  - `riskLevel`：风险等级（可选）
- 响应：
  - 返回 `downloadUrl`（推荐）或直接返回文件流（由前端通过 `blob` 下载）。

---

### 2. 导出按钮 & 下载反馈

**前端功能点**
- 点击“导出”按钮后：
  - 校验必填条件（如时间范围、报表类型）。
  - 显示“导出中”加载状态，防止重复点击。
  - 导出成功后，自动下载文件或提供下载链接。

**后端需要做的事**
- 在 `POST /api/report/export` 中：
  - 校验请求参数合法性（时间范围是否过大、是否有权限）。
  - 触发报表生成逻辑（同步生成或异步任务）。
  - 同步方式：直接返回二进制文件流。
  - 异步方式：立即返回任务ID，前端轮询任务状态并在完成后获取下载链接（可选增强）。
- 返回结构建议（同步简单版）：
  - `fileName`：建议的文件名（如 `患者清单_2025-01-01_2025-01-31.xlsx`）
  - `downloadUrl`：前端直接打开该 URL 下载。

---

### 3. 报表字段内容约定（以“患者清单”和“随访记录报表”为例）

**前端功能点**
- 报表字段本身在 Excel / CSV 里展示，前端主要负责：
  - 把当前页面上已选条件传给后端
  - 比如勾选“导出当前筛选条件下的患者”。

**后端需要做的事**
- 患者清单报表（`reportType = patientList`）：
  - 字段示例：患者编号、姓名、性别、年龄、证型、主要病种、风险等级、责任医生、最近随访时间等。
- 随访记录报表（`reportType = followup`）：
  - 字段示例：患者姓名、随访日期、随访方式（电话/上门/门诊）、主要结论、风险评估结果、下次随访时间等。
- 只要后端确保字段在报表中完整输出，前端不需要逐字段定义，只需知道“这一类报表大概包含哪些列”，方便在页面上写一句说明文字。

## 四、患者管理（路由：`/patient`，组件：`views/patient/List.vue` 等）

> 说明：本节包括“患者列表页”和“患者档案与画像详情页”两个核心页面。

### 1. 患者列表页（`/patient`）

**前端功能点**
- 列表展示：患者姓名、性别、年龄、证型/主病、风险等级、责任医生、最近随访时间、状态等。
- 筛选与搜索：
  - 关键字搜索（姓名、手机号、身份证号等）
  - 按风险等级、主病种、责任医生、标签（如重点关注）等筛选
  - 支持分页、每页数量切换。
- 列表操作：
  - 点击行进入患者详情 `/patient/detail/:id`
  - 快捷入口：新建患者档案按钮。

**后端需要做的事**
- 提供患者列表接口（示例）：
  - `GET /api/patient/summary`
- 请求参数建议：
  - `page` / `pageSize`
  - `keyword`（模糊匹配姓名 / 手机 / 身份证）
  - `riskLevel`、`mainDisease`、`doctorId`、`tag` 等筛选项
- 响应字段建议：
  - `total`：总条数
  - `rows: PatientSummaryData[]`，每条包含：
    - `id`、`name`、`gender`、`age`
    - `mainDisease`、`syndrome`、`riskLevel`
    - `doctorName`、`lastFollowTime`
    - `status`（在管/出院/失访 等）

---

### 2. 患者档案与画像详情页（`/patient/detail/:id`）

**前端功能点**
- 基本信息卡片：
  - 展示姓名、性别、年龄、联系方式、住址、主病种、证型等基础信息。
- 诊断与证型卡片：
  - 展示西医诊断、中医证候、疾病阶段、合并症等。
- 补充信息卡片：
  - 展示紧急联系人、病案号、入院出院时间、医保类型等补充字段。
- 画像与风险分析区：
  - 展示生活方式（饮食、运动、作息）、体质分类、依从性、历史随访结论、风险趋势概览等。
- 页面内跳转/操作：
  - 从档案页跳转到随访记录列表、复查计划、告警记录等相关模块。

**后端需要做的事**
- 提供患者详情接口（示例）：
  - `GET /api/patient/detail/:id`
- 建议分块返回数据，方便前端按卡片展示：
  - `basicInfo`：基础信息（姓名、性别、年龄、联系方式、地址、主病、证型等）
  - `diagnosis`：诊断与证型（西医诊断列表、中医证候列表、合并症等）
  - `extraInfo`：补充信息（紧急联系人、病案号、入出院时间、医保信息等）
  - `profile`：画像信息（体质、生活方式标签、依从性评分等）
  - `riskSummary`：风险与随访概览（当前风险等级、近几次随访结论、小结文字等）
- 若页面上有“编辑/保存档案”功能：
  - 需提供 `PUT /api/patient/detail/:id` 或类似接口，接收前端修改后的字段并持久化。

## 五、随访管理（路由：`/followup` 相关）

> 说明：本节涵盖“随访工作台”、“随访人员管理”、“复查计划管理”等与随访任务相关的页面。

### 1. 随访工作台（示例路由：`/followup/workbench`）

**前端功能点**
- 列表或看板形式展示当前待执行、进行中、已完成的随访任务。
- 支持按时间范围、责任随访人员、随访方式（电话/上门/门诊）、风险等级等筛选。
- 支持分页、跳转到患者档案与随访记录详情。
- 在任务上提供“开始随访 / 完成随访 / 重新预约”等快捷操作按钮。

**后端需要做的事**
- 提供随访任务列表接口（示例）：
  - `GET /api/followup/tasks`
- 请求参数建议：
  - `page` / `pageSize`
  - `status`（待随访/进行中/已完成 等）
  - `staffId`、`doctorId`、`riskLevel`、`followMethod`、`dateRange` 等筛选条件
- 响应字段建议：
  - `total`：总条数
  - `rows[]`：每条任务包括：
    - `taskId`、`patientId`、`patientName`
    - `riskLevel`、`mainDisease`
    - `planTime`（计划随访时间）、`finishTime`（完成时间）
    - `followMethod`（电话/上门/门诊）、`status`、`staffName` 等。
- 对应操作接口：
  - `POST /api/followup/task/start`（开始随访）
  - `POST /api/followup/task/complete`（完成随访并保存结论）
  - `POST /api/followup/task/reschedule`（重新预约时间）

---

### 2. 随访人员管理（`/followup/staff`，组件：`views/followup/StaffList.vue`）

**前端功能点**
- 表格展示随访人员列表：姓名、工号/账号、所属机构/科室、角色、当前任务数、状态等。
- 支持搜索（姓名/账号）、按机构/科室/状态筛选、分页。
- 支持在页面中弹窗“注册随访人员”，录入基本信息（姓名、工号/账号、联系方式、所属机构科室、角色等），提交后新增到列表。

**后端需要做的事**
- 提供随访人员列表接口：
  - `GET /api/followup/staff`
- 请求参数建议：
  - `page` / `pageSize`
  - `keyword`（姓名/账号）
  - `orgId` / `deptId` / `status` 等筛选条件
- 响应字段建议：
  - `total`
  - `rows[]`：每条包含 `id`、`name`、`jobNo`、`mobile`、`orgName`、`deptName`、`role`、`status`、`currentTaskCount` 等。
