# 智能随访系统——数据库结构设计说明

## 1. 文档目的
本文档用于详细描述“智能随访”系统当前版本的数据库结构设计，包括：

- 表的职责划分与命名规范
- 主子表（父子表）关系与外键约束策略
- 关键字段与通用字段（ext1~ext5）约定
- 索引设计与性能考虑
- 与后端接口（/patient/**、/files/**、随访相关接口）的对应关系

> 说明：本文档对应项目 `src/main/resources/schema.sql` 的设计与实现。

---

## 2. 设计原则

## 2.1 命名规范
- 表名：使用 `snake_case`，业务域前缀明确，例如 `patient_*`、`follow_up_*`。
- 字段名：使用 `snake_case`。
- 约束名：
  - 外键：`fk_<table>_<ref>`
  - 唯一键：`uk_<table>_<field>`
  - 索引：`idx_<table>_<fields>`

## 2.2 主键策略
- 所有表主键统一为 `BIGINT` 自增（`AUTO_INCREMENT`），字段名为 `id`。

## 2.3 严格参照完整性（外键策略）
系统采用“严格约束”策略：
- **所有明确的主子表关系均使用外键（FOREIGN KEY）约束**。
- 外键策略统一为：
  - `ON UPDATE CASCADE`
  - `ON DELETE RESTRICT`

含义：
- 禁止删除仍被引用的主记录，避免子表产生孤儿数据。
- 如需“主记录删除时级联删除子表数据”，需将特定外键改为 `ON DELETE CASCADE`（该策略需要结合业务审慎使用）。

## 2.4 通用字段与扩展字段
### 2.4.1 created_at / updated_at
- `created_at`：记录创建时间
- `updated_at`：记录更新时间

### 2.4.2 ext1 ~ ext5（预留扩展字段）
为满足未来需求变更，每张表统一预留 5 个扩展字段：
- `ext1` ~ `ext5`：`VARCHAR(255)`

约定：
- 默认不承载核心业务逻辑，仅用于“临时扩展”“灰度字段”“配置透传”等场景。
- 若扩展字段被长期使用，应在后续版本中升级为显式业务字段，并补齐索引与约束。

## 3. 核心实体与关系概览

## 3.1 用户（user）作为系统主体表
系统把 `user` 作为身份主体表，承载：
- 患者（PATIENT）
- 医生（DOCTOR）
- 随访人员（FOLLOW_UP）

大量业务表通过 `patient_id / doctor_id / staff_id / follow_up_user_id / uploader_user_id` 等字段引用 `user.id`。

## 3.2 主子表（父子表）关系示例
- **患者（user） → 患者基础信息（patient_basic_info）**：1 对 1
- **患者（user） → 每日测量（patient_daily_measurement）**：1 对多
- **患者（user） → 消息（patient_message）**：1 对多
- **随访任务（follow_up_task） → 随访日程（follow_up_schedule）**：1 对多

---

## 4. 表结构说明（按业务域）

## 4.1 账号与身份域

### 4.1.1 user
**用途**：系统统一身份主体表。

**关键字段**：
- `id`：主键
- `username/password`：登录凭证（若启用账号密码）
- `role`：角色（PATIENT/DOCTOR/FOLLOW_UP 等）
- `risk_level`：患者风险等级
- `status`：启用状态

**被引用关系**：
- 作为患者：被 `patient_*` 表通过 `patient_id` 引用
- 作为医生：被 `follow_up_plan.created_by_doctor_id`、`patient_followup_record.doctor_id` 等引用
- 作为随访人员：被 `follow_up_task.follow_up_user_id`、`patient_followup_record.staff_id` 等引用

---

## 4.2 随访业务域（follow_up_*）

### 4.2.1 follow_up_task
**用途**：随访任务主表，描述一次随访任务的触发、指派与状态。

**关键字段**：
- `patient_id`：患者 ID（FK -> user.id）
- `doctor_id`：医生 ID（FK -> user.id）
- `follow_up_user_id`：随访人员 ID（FK -> user.id）
- `trigger_type`：触发类型
- `status`：任务状态

**外键**：
- `fk_follow_up_task_patient`
- `fk_follow_up_task_doctor`
- `fk_follow_up_task_follow_up_user`

**索引**：
- `patient_id`、`doctor_id`、`follow_up_user_id` 单列索引
- `(status, created_at)` 联合索引用于按状态+时间分页

### 4.2.2 follow_up_plan
**用途**：随访计划（频率、周期等）。

**关键字段**：
- `patient_id`：患者 ID（FK -> user.id）
- `frequency_type/frequency_value`：频率配置
- `created_by_doctor_id`：创建医生（FK -> user.id）

**外键**：
- `fk_follow_up_plan_patient`
- `fk_follow_up_plan_created_by_doctor`

### 4.2.3 follow_up_schedule
**用途**：随访日程表（计划生成的每一次应随访点）。

**关键字段**：
- `patient_id`：患者 ID（FK -> user.id）
- `plan_id`：计划 ID（FK -> follow_up_plan.id）
- `related_task_id`：关联任务（FK -> follow_up_task.id）
- `due_at`：应随访时间
- `status`：完成状态

**索引**：
- `(patient_id, due_at)`：按患者的日程列表
- `(due_at, status)`：按时间窗口统计/提醒

## 4.3 健康监测与告警域

### 4.3.1 health_metric
**用途**：健康指标测量记录（可扩展多指标类型）。

**关键字段**：
- `patient_id`（FK -> user.id）
- `metric_type`：指标类型（血压/血糖/体温等）
- `value1/value2`：数值（例如血压可用双值）
- `measured_at`：测量时间
- `created_by_user_id`：记录创建人（FK -> user.id）

### 4.3.2 health_alert
**用途**：健康指标触发的告警记录。

**关键字段**：
- `patient_id`（FK -> user.id）
- `severity/status`：告警等级/处理状态

### 4.3.3 device_alert
**用途**：设备上报的告警事件。

**关键字段**：
- `patient_id`（FK -> user.id）
- `device_sn/alert_code/alert_message`
- `occurred_at`

### 4.3.4 revisit_record
**用途**：复诊记录。

**关键字段**：
- `patient_id`（FK -> user.id）
- `created_by_doctor_id`（FK -> user.id）
- `revisit_at/hospital/department/notes`

### 4.3.5 risk_level_history
**用途**：风险等级历史变更记录。

**关键字段**：
- `patient_id`（FK -> user.id）
- `risk_level/assessed_at/source`

### 4.3.6 syndrome_assessment
**用途**：证候评估记录。

**关键字段**：
- `patient_id`（FK -> user.id）
- `created_by_user_id`（FK -> user.id）
- `syndrome_code/syndrome_name/score/assessed_at`

## 4.4 小程序患者端域（patient_*）

> 这些表主要用于 `/patient/**` 系列接口。

### 4.4.1 patient_basic_info
**用途**：患者基础信息（1 对 1）。

**关键字段**：
- `patient_id`：唯一（UK），同时外键引用 `user.id`
- `height_cm/baseline_weight_kg/birthday/id_card`
- `emergency_contact_name/emergency_contact_phone/insurance_type`

### 4.4.2 patient_daily_measurement
**用途**：患者日常测量上报（1 对多）。

**关键字段**：
- `patient_id`（FK）
- `measured_at`：测量时间
- `sbp/dbp/heart_rate/weight_kg/temperature_c/spo2`
- `symptoms`

**索引**：
- `(patient_id, measured_at)`：按患者与时间排序查询

### 4.4.3 patient_lab_result
**用途**：化验/检验结果记录。

**关键字段**：
- `patient_id`（FK）
- `lab_type`：化验类型
- `value_text/value_num/unit/status/measured_at/hospital/remark`

### 4.4.4 patient_rehab_task
**用途**：患者康复/康养任务。

**关键字段**：
- `patient_id`（FK）
- `title/description/difficulty/status/start_date/due_date/completed_at`

### 4.4.5 patient_tcm_survey
**用途**：中医问卷记录。

**关键字段**：
- `patient_id`（FK）
- `survey_type`
- `answers_json/result_json`

**类型说明**：
- `answers_json/result_json` 采用 `TEXT`，以兼容前端传入“非严格 JSON 文本”。

### 4.4.6 patient_linked_account
**用途**：患者关联账号（家属/监护人/其他账号）关系表。

**关键字段**：
- `patient_id`（FK -> user.id）
- `linked_user_id`（FK -> user.id）
- `relation/linked_username/linked_phone/status`

### 4.4.7 patient_message
**用途**：患者消息中心。

**关键字段**：
- `patient_id`（FK）
- `title/content/message_type/status/read_at`

### 4.4.8 patient_feedback
**用途**：患者反馈。

**关键字段**：
- `patient_id`（FK，可为空但建议业务上必填）
- `feedback_type/content/contact/status/reply/replied_at`

### 4.4.9 patient_followup_record
**用途**：患者随访记录（医生/随访人员记录一次随访详情）。

**关键字段**：
- `patient_id`（FK -> user.id）
- `doctor_id`（FK -> user.id）
- `staff_id`（FK -> user.id）
- `followup_time/followup_type/result_status/risk_level`
- 血压/心率/体重等生命体征字段
- 中医相关字段（舌像、脉象、结论等）
- `next_followup_date/next_followup_type`

### 4.4.10 patient_followup_application
**用途**：患者发起随访申请。

**关键字段**：
- `patient_id`（FK -> user.id）
- `preferred_time/contact_phone/apply_reason`
- `status`
- `handled_by_user_id`（FK -> user.id）

## 4.5 文件域

### 4.5.1 file_upload
**用途**：文件上传记录表，用于统一管理上传文件元数据。

**关键字段**：
- `biz_type`：业务类型（用于区分舌像/报告/头像等）
- `patient_id`（FK -> user.id）
- `uploader_user_id`（FK -> user.id）
- `original_name/file_name/file_ext/mime_type/file_size`
- `storage_path`：服务器本地保存路径
- `url`：对外访问路径（通常为 `/uploads/...`）
- `sha1`：文件指纹（去重/校验）
- `status`

## 6. 运维与落库建议

## 6.1 外键启用注意事项
在已有历史数据的数据库上执行新增外键约束时，可能因历史脏数据失败：
- 子表存在 `patient_id` 在 `user` 不存在
- 外键列类型不一致
- 外键列为 NULL（但声明为 NOT NULL）

建议顺序：
1. 先导入主表（user）
2. 再导入依赖子表数据
3. 最后启用外键（或确保一开始就以外键约束建库）

## 6.2 索引与性能
- 所有外键列均应具备索引，以提升联表查询与约束检查性能。
- 时间序列类表（如测量、随访记录）建议以 `(patient_id, time)` 作为主要查询索引。

## 6.3 表注释与字段注释（COMMENT）维护方式
系统支持在 MySQL 中为：

- **字段（列）** 增加注释：`COMMENT '...'`
- **表** 增加注释：`ALTER TABLE <table> COMMENT='...'`

### 6.3.1 方案选择
本项目采用可在“表已存在”场景下仍可生效的方式（方案 B）：

- 字段注释：在 `schema.sql` 末尾追加 `ALTER TABLE ... MODIFY COLUMN ... COMMENT '...'`
- 表注释：在 `schema.sql` 末尾追加 `ALTER TABLE ... COMMENT='...'`

### 6.3.2 执行顺序建议
推荐执行顺序如下：

1. 执行 `CREATE TABLE ...`（新建库时）
2. 执行末尾的列注释 `ALTER TABLE ... MODIFY COLUMN ... COMMENT ...`
3. 执行末尾的表注释 `ALTER TABLE ... COMMENT=...`

### 6.3.3 注意事项
- `CREATE TABLE IF NOT EXISTS` 对已存在表不会重建，因此仅靠 `CREATE TABLE` 无法更新注释。
- `MODIFY COLUMN` 会要求列定义（类型/可空/默认值）与现有表一致，否则可能报错；因此建议在“结构已一致”的前提下执行注释 SQL。
- 若数据库存在历史脏数据，外键约束可能导致建表或加约束失败，应先清理数据或分步骤执行。

## 7. 未来扩展建议
- 若后续需要报表或复杂统计，可考虑引入视图（VIEW）或物化汇总表。
- ext1~ext5 如被长期使用，应升级为显式字段并补齐索引与约束。
